How the whole system works — end to end

A single walkthrough of TradeEngine (a Nepal-based MERN e-commerce platform): from a browser request hitting Cloudflare, through Nginx and Docker, into the Express API, the Redis/Bull order queue, the Fonepay QR payment gateway, and the Upaya delivery API — plus auth, real-time notifications, and the CI/CD that ships it all.

Frontend React 19 + Vite + Tailwind Backend Node 20 + Express (ESM) DB MongoDB + Mongoose Queue Redis 7 + Bull Realtime Socket.io + SSE Infra Docker · GHCR · VPS · Nginx · Cloudflare Domain tradengine.com.np

The one-paragraph story (say this first)

Elevator pitch "It's a MERN e-commerce platform for the Nepali market. The React SPA and the Express API each run in their own Docker container behind Nginx, with Redis for a job queue and MongoDB for data. Cloudflare sits at the edge for DNS, TLS and caching. When a customer places an order, the request goes through an Nginx reverse proxy into Express, which drops the order onto a Redis/Bull queue processed one job at a time — that serialization is what prevents overselling the last unit of stock. Payment is handled by Fonepay's dynamic-QR gateway and delivery is booked automatically on the Upaya courier API. Everything ships through a GitHub Actions CI/CD pipeline that builds Docker images, pushes them to GHCR, and SSHes into the VPS to pull and restart."

The three flows an interviewer will drill into

⚙ Order Queue

Why a queue at all? Concurrency = 1 serializes stock decrements so two people can't both buy the last item. Backed by Redis, wrapped in a Mongo transaction. → Section 5

₨ Fonepay

Dynamic QR + oAuth with RSA request signing. Two settlement paths (WebSocket + polling), idempotent, amount-verified. → Section 6

🚚 Upaya Delivery

Auto-books a courier shipment on status change, then maps courier tracking back onto our order states. → Section 7

Step by step

1
Browser resolves tradengine.com.np
DNS is managed by Cloudflare, so the domain resolves to a Cloudflare edge IP — never your VPS IP directly. Your origin server is hidden behind Cloudflare.
2
Cloudflare edge TLS · CDN · WAF
Cloudflare terminates HTTPS (TLS cert at the edge), serves cached static assets from its CDN, applies WAF/DDoS protection and rate limiting, then forwards the request to your VPS origin over its network.
3
VPS host Nginx — reverse proxy
On the VPS, a host-level Nginx receives the proxied traffic and forwards it to the frontend container published on port 3000. (This is the deployment topology; the container-level Nginx config below is the one that lives in the repo.)
4
Frontend container Nginx nginx.conf
This Nginx does three jobs: (a) serves the built React SPA static files, (b) SPA fallback routing to index.html, and (c) proxies every /api/* call to the API container at http://api:5000. It also special-cases /product/* for social crawlers (Open Graph link previews).
5
Express API container :5000
Node/Express receives /api/v1/*. Global middleware runs first (helmet, CORS allowlist, JSON body parse, cookie-parser, mongo-sanitize, hpp), then the request hits the matching route → controller.
6
Data layer: MongoDB + Redis
Controllers read/write MongoDB via Mongoose. Order placement additionally pushes a job to Redis (Bull). All three app containers share a private Docker bridge network ecommerce-net, so they reach each other by service name (api, redis).
E-Commerce-Frontend/nginx.conf — the container proxy
server {
  listen 3000;

  # 1) Serve the SPA; fall back to index.html for client-side routes
  location / {
    try_files $uri $uri/ /index.html;
  }

  # 2) Proxy the API to the backend container (service name "api")
  location /api/ {
    proxy_pass http://api:5000/api/;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_pass_header Set-Cookie;   # keep httpOnly refresh cookie
  }
}

services:

    image: ghcr.io/${GHCR_OWNER}/ecommerce-frontend:latest
    ports: ["3000:3000"]
  api:       # Node/Express
    image: ghcr.io/${GHCR_OWNER}/ecommerce-api:latest
    ports: ["5000:5000"]
    environment: { REDIS_HOST: redis, NODE_ENV: production }
    depends_on: { redis: { condition: service_healthy } }

Why Docker service names work as hostnames: Compose puts all three containers on one bridge network and runs an internal DNS, so http://api:5000 and redis:6379 resolve container-to-container without exposing anything publicly.
If asked "why two Nginx layers?" "The container's Nginx is part of the frontend image — it serves the SPA and knows how to reach the API container. The host Nginx + Cloudflare handle the public edge: TLS, the real domain, and routing to the container. Separating them keeps the image portable — it runs the same locally or on any host."
Section 2 · DevOps

CI/CD: from git push to live in production

One GitHub Actions workflow, triggered on push to main, does everything: build images, push to the registry, deploy to the VPS, and clean up. Three jobs, run in order.

1
Trigger — push to main .github/workflows/deploy.yml
Any commit to main (or a manual workflow_dispatch) starts the pipeline on a GitHub-hosted ubuntu-latest runner.
2
Job 1 — Build & Push Docker images
Uses Docker Buildx to build two images (API + frontend) from their Dockerfiles, then pushes them to GHCR (GitHub Container Registry) tagged both :latest and :<git-sha>. Layer cache (type=gha) makes rebuilds fast. The frontend build bakes in VITE_API_BASE_URL and the Google client ID as build args.
3
Job 2 — Deploy over SSH needs: build-and-push
Using appleboy/ssh-action with a private key from secrets, it SSHes into the VPS and runs: git reset --hard origin/main (sync compose files) → docker login ghcr.io → docker compose pull → docker compose up -d → docker image prune -a.
4
Job 3 — Prune old GHCR versions
Deletes old container image versions from GHCR, keeping only the latest 2 of each — so the registry doesn't grow forever.
.github/workflows/deploy.yml — the deploy step on the VPS
script: |
  set -e
  # 1. Server mirrors origin/main exactly (VPS edits are discarded)
  cd ~/E-commerce-vps
  git fetch origin main && git reset --hard origin/main
  # 2. Authenticate to the registry
  echo "$GHCR_TOKEN" | docker login ghcr.io -u "$GH_ACTOR" --password-stdin
  # 3. Pull freshly-built images, 4. rolling restart (keeps Redis data)
  docker compose -f docker-compose.prod.yml pull
  docker compose -f docker-compose.prod.yml up -d --remove-orphans
  # 5. Reclaim disk
  docker image prune -a -f

🔐 Where secrets live

VPS_HOST, VPS_USER, SSH_PRIVATE_KEY, VITE_GOOGLE_CLIENT_ID are GitHub Actions secrets. GHCR auth reuses the auto-provided GITHUB_TOKEN — no PAT to manage.

♻️ Zero-downtime-ish

up -d only recreates containers whose image changed. Redis data survives via the named volume, so the queue and cache aren't wiped on deploy.

The clean summary "Push to main triggers GitHub Actions. It builds the two Docker images, pushes them to GHCR tagged with the commit SHA, then SSHes into the VPS to pull those images and docker compose up -d. It's fully automated — I don't touch the server. The SHA tags mean I can roll back to any previous build instantly."
Gotcha to mention: git reset --hard on the server means the VPS is a pure mirror of main — never hand-edit files on the box, they'll be blown away on the next deploy. That's intentional: the repo is the single source of truth.
Section 3 · Security

Authentication & sessions — the JWT dance

Two tokens, two lifetimes, two storage locations. The access token authorizes API calls; the refresh token silently renews it. Staff accounts get an extra single-session lock.

A Access token

Short-ish JWT. Stored in localStorage, sent on every request as Authorization: Bearer …. Verified by the protect middleware. Lifetime: 1 day for staff, 30 days for customers.

R Refresh token

JWT stored in an httpOnly cookie (JS can't read it → XSS-safe). Only used at /auth/refresh-token to mint a new access token. Also saved in the user doc so it can be revoked.

Login → authenticated request → silent refresh

1
Login POST /auth/login
Password checked with bcrypt. On success, generateTokenPair() mints an access + refresh token that both embed a random sid (session id). The refresh token is set as an httpOnly cookie; the access token is returned in the JSON body.
2
Every API call carries the access token
The Axios request interceptor attaches Bearer <accessToken>. On the server, protect verifies the JWT signature, loads the user, and rejects if blocked or (for staff) if the session was replaced.
3
Access token expires → 401
The Axios response interceptor catches the 401, pauses the failed request, and calls POST /auth/refresh-token — the browser auto-sends the httpOnly cookie (withCredentials).
4
New access token → replay original request
Server validates the refresh cookie against the stored token, mints a fresh access token (same sid), returns it. The interceptor stores it and replays the original request transparently — the user never notices. Concurrent 401s are queued so only one refresh fires.
frontend src/api/client.js — the auto-refresh interceptor (trimmed)
client.interceptors.response.use(res => res, async error => {
  if (error.response?.status === 401 && !original._retry) {
    original._retry = true;
    // refresh token rides in the httpOnly cookie — we never see it in JS
    const { data } = await axios.post(`${BASE_URL}/auth/refresh-token`, {}, { withCredentials: true });
    localStorage.setItem('accessToken', data.data.accessToken);
    original.headers.Authorization = `Bearer ${data.data.accessToken}`;
    return client(original);   // replay the original request
  }
  return Promise.reject(error);
});

Single-session lock for admin / employee

Every staff login rotates a random activeSessionId stored on the user. Both tokens embed the current sid. On each request, protect compares the token's sid to the DB's activeSessionId — if a staff account logs in elsewhere, the old session's sid no longer matches and it's force-logged-out with a SESSION_REPLACED tag the frontend recognizes.

backend src/middleware/auth.middleware.js — the session check
const isStaff = user.role === "admin" || user.role === "employee";
if (isStaff && user.activeSessionId && decoded.sid !== user.activeSessionId) {
  throw new ApiError(401, "SESSION_REPLACED: signed in from another location");
}

Likely question "Why store the refresh token in a cookie but the access token in localStorage?" — Answer: the refresh token is the long-lived, high-value credential, so it goes in an httpOnly cookie that JavaScript literally cannot read, defeating XSS token theft. The access token is short-lived and needs to be attached to the Authorization header by JS, so it lives in localStorage; even if stolen it expires and can't mint new tokens.
Google login uses google-auth-library to verify the ID token server-side, then issues the same token pair — so downstream everything is identical to password login.

Inside placeOrder — everything that happens before the queue

1
Resolve the buyer's intent POST /orders
Reads shippingAddressId + paymentMethod (COD / ONLINE). Supports two paths: cart checkout (useCart) or Buy Now (a single directItem without touching the cart).
2
Re-price every line item server-side
For each item it re-fetches the live product, resolves the chosen color variant (which drives price, stock and image), and rejects if stock is insufficient. Prices are never trusted from the client — the server recomputes itemsPrice.
3
Compute shipping — 3-tier priority
(1) A custom DeliveryArea matching the address city (admin override), else (2) a live Upaya rate quote if the address has an Upaya location id, else (3) the admin's default deliverySettings (flat charge, free over a threshold).
4
Apply coupon (re-validated, server-side)
Validity, audience, and per-item eligibility are all re-checked. Handles FREEBIE (adds a Rs.0 line + reserves its stock) and FREE_SHIPPING (zeroes the shipping). Coupon usage isn't committed here — that happens atomically inside the queue job.
5
COD rules + booking advance
For COD: checks min/max order limits and whether COD is enabled. If a booking advance is configured, it computes the deposit (flat or %) and marks codBookingStatus = PENDING — the advance is later collected via a Fonepay QR.
6
Hand off to the queue → wait for result
Builds jobData, adds it to the Bull queue, and await job.finished() — so the HTTP response still waits for the real result (success or out-of-stock), but the critical stock mutation runs serialized inside the worker.
backend src/controllers/order.controller.js — the handoff
if (process.env.SKIP_QUEUE === "true") {
  // dev: no Redis — run the processor inline
  result = await processOrderJob({ data: jobData });
} else {
  const { default: orderQueue } = await import("../queues/order.queue.js");
  const job = await orderQueue.add(jobData);   // enqueue
  result = await job.finished();          // wait for the worker's result
}

Order status pipeline (what the customer sees)

PLACED→ CONFIRMED→ PACKED→ SHIPPED→ OUT_FOR_DELIVERY→ DELIVERED   CANCELLED RETURNED

Each transition pushes to statusHistory, notifies the customer, and — from CONFIRMED onward — auto-dispatches to Upaya. Reaching DELIVERED also flips paymentStatus → PAID for COD. Customers can only cancel while PLACED; staff can cancel up to CONFIRMED (after that the courier shipment exists).

The framing that impresses "The controller does all the validation and pricing, but the one thing that mutates shared state — decrementing stock and creating the order — is pushed onto a serialized queue. That separation is deliberate: cheap read-heavy validation runs concurrently, and only the race-prone write is serialized."
Section 5 · The showpiece

Redis + Bull: why a queue prevents overselling

This is the single most impressive part of the system to explain well. The problem is a classic race condition; the solution is serialization via a single-concurrency worker, hardened with a database transaction.

The problem, in one sentence Two customers click "Buy" on the last unit at the same millisecond. Without coordination, both requests read stock = 1, both pass the check, both decrement → stock = -1, and you've sold an item you don't have.

The fix: a Bull queue with concurrency = 1

Every order becomes a job in a Redis-backed Bull queue. The worker processes one job at a time, so stock checks and decrements can never interleave. The second buyer's job runs after the first finished — sees stock = 0 — and gets a clean "out of stock" error.

backend src/queues/order.queue.js
const orderQueue = new Bull("order-queue", {
  redis: redisConfig,
  defaultJobOptions: {
    attempts: 3,                                // retry failed jobs 3×
    backoff: { type: "exponential", delay: 2000 }, // 2s, 4s, 8s…
    timeout: 30000,
    removeOnComplete: { count: 100 },
  },
});

// Concurrency = 1 → jobs run one at a time (serialized).
// THIS is what prevents the oversell race.
orderQueue.process(1, processOrderJob);

Inside the worker — processOrderJob

1
Open a MongoDB transaction (prod)
In production Mongo runs as a replica set, so the whole job runs in a transaction — either every write commits together or none do. (Dev standalone Mongo can't do transactions, so it falls back to manual rollback.)
2
Atomically verify + decrement each item's stock
A single findOneAndUpdate with the filter stock: {$gte: qty} and $inc: {stock: -qty}. Because the filter and update are one atomic op, the check-then-decrement can't be split. Color variants use $elemMatch + arrayFilters to hit the right color's stock.
3
If any item fails → roll back
If findOneAndUpdate returns null (insufficient stock), it throws with a precise message ("Available: 2, Requested: 3"). The transaction aborts, restoring every already-decremented item.
4
Create the order + inventory logs
Writes the Order document (status PLACED, paymentStatus PENDING) and an InventoryLog row per item for auditability.
5
Claim the coupon — guarded, exactly-once
A guarded findOneAndUpdate increments usedCount and adds the user to usedBy only if still valid and unused. Concurrent orders against a capped coupon → exactly one wins, the other is rolled back.
6
Clear cart, notify, commit
Empties the cart, writes an in-app notification (pushed live via SSE), then commits the transaction. After commit, a non-blocking setImmediate fans out staff notifications + low-stock alerts.
backend src/queues/order.processor.js — the atomic decrement
// Filter REQUIRES enough stock; update decrements it — one atomic op.
const filter = { _id: item.product, isDeleted: false, isPublished: true };
const inc = { sold: item.quantity, stock: -item.quantity };
if (item.color) {                       // colored line → target that color's stock
  filter.colors = { $elemMatch: { name: item.color, stock: { $gte: item.quantity } } };
  inc["colors.$[c].stock"] = -item.quantity;
  opts.arrayFilters = [{ "c.name": item.color }];
} else {
  filter.stock = { $gte: item.quantity };
}
const product = await Product.findOneAndUpdate(filter, { $inc: inc }, opts);
if (!product) throw new Error(`"${item.title}" is out of stock`); // → rollback

🛡️ Two layers of safety

1. Serialization (concurrency=1) means jobs don't overlap. 2. Atomic findOneAndUpdate means even the single write is race-proof. Belt and suspenders.

🔁 Automatic retries

Transient failures (a blip in Mongo) retry 3× with exponential backoff. A real out-of-stock error isn't retried into success — it surfaces to the user as a 409.

Anticipate the pushback: "Doesn't concurrency=1 kill throughput?" — Honest answer: it serializes only the final stock write, which is milliseconds of Mongo work; all the expensive validation/pricing already ran concurrently in the controller. For this store's volume it's more than enough, and it's the correct trade — correctness over raw throughput on the money path. If it ever became a bottleneck you'd shard the queue per-product or move to atomic per-document ops without the global lock.
One-liner to land it "The queue turns a concurrency problem into a sequencing problem. Instead of fighting a race, I removed the race — orders that touch stock simply can't run at the same time."
Nice detail: a SKIP_QUEUE=true env flag bypasses Redis entirely and runs the processor inline — handy for local dev where you don't want to run Redis.
Section 6 · Payments

Fonepay — dynamic-QR payments, done safely

Fonepay is Nepal's interbank QR network. This is a fully automatic flow (no uploaded screenshots): generate a single-use QR, the customer scans and pays from any bank/wallet app, and the order settles the moment Fonepay confirms — verified two independent ways.

💳 purpose: "full"

Collects the whole ONLINE order amount. Success flips paymentStatus → PAID.

🧾 purpose: "booking"

Collects the COD advance deposit. Success flips codBookingStatus → PAID and confirms the COD order.

The happy path, end to end

1
Request a QR POST /payments/fonepay/:orderId/qr
The controller checks the order belongs to the user and resolves the amount for the purpose. Then it asks the Fonepay service for a dynamic Intent QR.
2
Fonepay service: oAuth login (signed)
First call does an oAuth login — username/password with HTTP Basic auth plus an RSA signature over the exact request body. The returned bearer token (an encrypted JWE) is cached process-wide until just before it expires.
3
Generate Intent QR (signed)
Posts amount, billId (= order number), terminalId, and a unique referenceLabel (the PRN), again RSA-signed. Fonepay returns a QR string + a per-transaction WebSocket URL.
4
Render QR + start the live watcher
The QR string is rendered to a PNG data-URL server-side (via qrcode) so the frontend just shows an <img>. The PRN/terminal/status are saved on the order. A background WebSocket watcher connects to Fonepay's socket for instant settlement.
5
Customer scans & pays
They pay from any Fonepay-member bank or wallet app. Two things now race to detect success — and that's fine, because settlement is idempotent.
6
Settle — verified & idempotent
Both the WebSocket watcher and the frontend's status polling call settleFonepayPayment, which asks Fonepay's Get-Status API for the authoritative result, checks the paid amount matches, then does an atomic "claim" so only the first path credits the order.

Two settlement paths, one source of truth

⚡ Path A — WebSocket (fast)

fonepayWatcher.js opens Fonepay's per-transaction socket. On any terminal message it triggers settlement. Works even if the customer closes the tab. Purely an optimization — best-effort, failures are swallowed.

🔁 Path B — Polling (reliable)

While the QR is on screen, the frontend polls GET /payments/fonepay/:orderId/status. This is the reliable fallback / source of truth — if the socket push is missed, polling still settles it.

backend src/controllers/payment.controller.js — the idempotent claim
// Ask Fonepay for the authoritative status
status = await fonepayService.getPaymentStatus({ referenceLabel: txn.prn });

if (status.paymentStatus === "success") {
  // Anti-tamper: the amount Fonepay reports must match what we asked for
  const paid = Number(status.totalTransactionAmount);
  if (Math.abs(paid - txn.amount) > 0.5) return { status: "FAILED", reason: "amount_mismatch" };

  // Atomic claim — only the update that flips "not paid → paid" wins,
  // so the WebSocket and the poll can't BOTH credit the order.
  const claimed = await Order.findOneAndUpdate(
    { _id: order._id, paymentStatus: { $ne: "PAID" } },
    { $set: { paymentStatus: "PAID", paidAt: now, ...txnSet } },
    { new: true });
  if (!claimed) return { status: "SUCCESS" };  // lost the race — already settled
  // …record a Payment doc + notify customer & staff
}

Why the RSA signing matters

Every Fonepay call sends a Signature header = the RSA-SHA256 signature of the exact JSON bytes in the body, signed with your private key. Fonepay verifies it with your public key. This proves the request genuinely came from your merchant account and wasn't tampered with in transit — it's request-level integrity on top of TLS.

backend src/services/fonepay.service.js — signing every request
export const signPayload = (rawBody) => {
  const signer = crypto.createSign("RSA-SHA256");
  signer.update(rawBody, "utf8");   // sign the SAME bytes sent on the wire
  return signer.sign(privateKey(), "base64");
};

The safety net: auto-cancel unpaid orders

An ONLINE order sits in PENDING until paid. A background job (orderTimeout.job.js) sweeps every 60s: it sends one reminder at the halfway mark, and auto-cancels + restores stock if payment isn't completed within PENDING_ORDER_TIMEOUT_MIN (default 30 min). This stops abandoned unpaid orders from silently locking up inventory.

Idempotency is the theme here. Two detectors, retried polling, WebSocket reconnects — any of them can fire more than once. The atomic conditional update (paymentStatus != "PAID") guarantees the order is credited exactly once, no duplicate Payment docs, no double notifications.
If asked "how do you know the payment is real?" "We never trust the client or even the WebSocket push blindly. On any success signal we call Fonepay's Get-Status API server-to-server, verify the returned amount equals what we charged, and only then flip the order to PAID — inside an atomic claim so it happens once."
Section 7 · Delivery

Upaya — booking & tracking the courier

Upaya is a Nepali courier/logistics API. Our system uses it three ways: to quote a shipping rate at checkout, to create a delivery order when fulfilment starts, and to pull live tracking that auto-advances our own order status.

backend src/services/upaya.service.js — one authenticated client
// Every call carries the API key; this wrapper centralizes auth + errors.
const request = (path, opts) => fetch(`${baseUrl()}${path}`, {
  headers: { "X-API-Key": apiKey() }, ... });

export const upayaService = {
  getLocations(),                // cached 1h — cities/areas rarely change
  rate({ location_id, weight }), // POST /order-rates  → shipping quote
  addOrder(payload),             // POST /add-order    → create shipment
  trackOrder(ref),               // GET  /track-order  → live status
};

1 · Rate quote at checkout

When an address carries an Upaya location id and there's no admin override for that city, placeOrder calls upayaService.rate() to get the real delivery cost for the parcel weight — so the customer pays the actual courier rate. (Recall the 3-tier shipping priority from Section 4.)

2 · Auto-dispatch when fulfilment starts

1
Staff moves the order to CONFIRMED+
When an admin/employee advances an order to CONFIRMED, PACKED, SHIPPED, or OUT_FOR_DELIVERY, dispatchToUpayaForFulfillment fires.
2
Build the shipment payload
dispatchToUpaya assembles receiver name/phone/address, the Upaya area_id, rolled-up parcel initial_weight, a product description, and — crucially — the COD amount (the order total for COD, or 0 for prepaid ONLINE orders).
3
POST /add-order → store the reference
On success it saves upayaOrderRef / upayaTrackingId, sets upayaSyncStatus = SYNCED, and bumps the order to SHIPPED with the tracking ref. It's idempotent — if already synced it returns the existing ref instead of double-booking.
4
Failures are recoverable
If Upaya errors, the failure is stored in upayaError with status FAILED — the customer flow is never blocked. Admin can hit a retry endpoint to re-dispatch after fixing the area.

3 · Live tracking → auto-advancing our status

When anyone opens the tracking view, getUpayaTracking calls trackOrder(ref), then maps Upaya's free-text status onto our clean enum and advances the order only forward (never backward, never past CANCELLED/RETURNED).

Why the mapping layer exists: third-party APIs return messy, inconsistent status text ("PICKED", "picked up", "IN_TRANSIT"). Normalizing to our own enum means the rest of the app — UI, notifications, analytics — only ever deals with 6 clean states, and swapping couriers later only touches this one mapping table.
Clean summary "Upaya is abstracted behind a single service wrapper. We quote rates at checkout, auto-create the shipment when staff confirm the order, and translate the courier's tracking back into our own status enum so the customer sees one consistent journey — with idempotent dispatch and a retry path if the courier call fails."
Section 8 · Real-time

Live layer — SSE notifications + Socket.io chat

Two different real-time mechanisms, chosen deliberately for two different shapes of problem: one-way server push vs. two-way conversation.

→ SSE — Notifications

Server-Sent Events = one-way, server→browser, over plain HTTP. Perfect for "your order shipped" pushes. Lighter than a WebSocket when you only ever push down.

⇄ Socket.io — Chat

WebSocket = full duplex. Needed for customer↔support chat: typing indicators, online presence, seen receipts — all bidirectional.

SSE notification flow (and the neat security trick)

1
Get a single-use ticket POST /notifications/stream-ticket
EventSource can't send an Authorization header, and putting a JWT in the URL leaks it to logs/proxies. So the client first makes an authenticated POST to exchange its token for a short-lived, single-use ticket.
2
Open the stream with the ticket GET /notifications/stream?ticket=…
The server validates the ticket (once), then registers this connection's res object against the user id in an in-memory map and holds the connection open as text/event-stream.
3
Server pushes events
Anywhere in the code, pushToUser(userId, payload) writes data: {…}\n\n to that user's open connections. Order placed, payment received, status changed → instant toast in the browser, no polling.
backend src/utils/sseClients.js — the push registry
const clients = new Map();  // userId → Set<res>

export function pushToUser(userId, payload) {
  const set = clients.get(userId.toString());
  if (!set) return;
  for (const res of set) res.write(`data: ${JSON.stringify(payload)}\n\n`);
}

Socket.io chat

On connect, a Socket.io middleware verifies the JWT from the handshake. An in-memory onlineUsers map tracks userId → socketId. Events: message:send persists to Mongo then emits message:receive to the recipient if online, plus message:seen, typing:start/stop, and user:online/offline presence broadcasts. It shares the exact same CORS allowlist as the REST API.

Classic question "SSE vs WebSocket — when would you use each?" — "SSE for one-way server→client push: simpler, runs over plain HTTP, auto-reconnects, ideal for notifications. WebSocket when you need bidirectional real-time — chat, presence, typing. I used SSE for notifications and Socket.io for chat precisely because their communication shapes differ."

★ Rapid-Fire

Interview cheat sheet

Short, confident answers to the questions that actually get asked. Read these the morning of.

"Walk me through what happens when a user places an order."

Browser → Cloudflare → Nginx → Express placeOrder. It validates the address, re-prices items server-side, computes shipping (custom area → Upaya rate → default), re-validates the coupon, checks COD rules — then enqueues a Bull job. The worker runs one job at a time inside a Mongo transaction: atomically decrements stock, creates the Order, logs inventory, claims the coupon, clears the cart, and notifies the user over SSE. If it's ONLINE, we generate a Fonepay QR; once paid (verified via Fonepay's status API) the order confirms and auto-dispatches to Upaya for delivery.

Rapid Q&A
