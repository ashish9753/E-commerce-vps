1. Introduction 
1.1. Purpose of the document 
This document outlines the Checkout Intent Flow , detailing the steps and logic governing the 
checkout process. It serves as a guide for stakeholders to understand the intent -based checkout 
mechanism, ensuring seamless user experience and efficient transaction handling.  
1.2. Scope of this Document 
o Defining the intent-driven approach for the checkout process. 
o Identifying key stages and decision points within the flow. 
o Outlining user interactions and system responses. 
o Highlighting dependencies and integration requirements. 
2. What is Deep Link? 
Deep linking is a powerful tool in mobile app development, allowing links to open specific content 
or screens within an app rather than just its home screen. This capability makes user navigation 
faster and more efficient, which is especially useful for ma rketing, notifications and ecom 
checkout. 
3. User Journey Overview 
Intent Checkout 
Step 1: User goes to the checkout section of the e-commerce platform. 
Step 2: User clicks on "Pay with Fonepay". 
Step 3: User selects "Pay via Mobile Banking App". 
Step 4: The system automatically fetches and displays all banks listed on the mobile phone.  
Step 5: The user selects their bank from the displayed list. 
Step 6: The user is redirected to the banking app. 
Step 7: The user opens the app, selects their "Debit account", and approves or rejects the payment. 
Step 8: Upon approval, the success payload is sent to the merchant’s website/app. If rejected or 
timed out, a failure payload is sent. 
4. Deep Linking in Android Checkout 
Deep linking is implemented in Android checkout to provide a seamless transition between the e -
commerce App/Web and the mobile banking app. When a user selects "Pay via Mobile Banking 
App," the system identifies the banking app installed on the device and redirects the user 
accordingly in the intent flow.

5.1. How WEB-to-App Communication Works? 
 
 
 
❖ Merchant Web Initiates Banking App 
• The Merchant’s Web application initiates the Banking App via a deep link.  
• It passes a qrPayload (containing essential transaction details like amount, merchant 
ID, reference ID, etc.) to the Banking App. 
❖ Banking App Processes the qrPayload 

===== PAGE 6 =====
 External 
 
Checkout Intent Flow  V1.10  6 
 
• The Payment App: 
o Parses and validates the qrPayload contents. 
o The user authenticates and confirms the payment transaction. 
 
❖ Real-time Status Update via WebSocket 
• Before the transaction begins: 
o A WebSocket connection is already established between the Merchant Web 
Client and the Fonepay QR WebSocket Server. 
• After the transaction is processed: 
o The Fonepay QR WebSocket Server sends the transaction result (success/failure 
with remarks) to the Merchant Web through this socket. 
 
❖ Status API Call 
• In addition to the WebSocket message: 
o The Merchant Backend (server-side) calls the Fonepay Backend Status API. 
o This API call double-checks and verifies the final status officially. 
❖ Final Status and Remarks Delivery 
• The Merchant Web displays the outcome based on: 
o WebSocket status message, and/or 
o Verified API status response. 
• Based on the results: 
o Success: Show success confirmation and proceed. 
o Failure: Show error message, possibly allow retry based on received remarks.

❖ Fully Flutter-based apps (using auto_route or go_router) 
❖ Fully Native Android apps  
❖ Flutter apps not using AppRouter or GoRouter

Signature Generation Logic: 
• Algorithm: Use SHA256withRSA. 
• Payload Construction: The signature must be generated using the entire request JSON 
body as the payload. 
• Key Usage: The client signs the payload using their PKCS8 private key (provided in Hex or 
Base64 format). Fonepay verifies this using the client’s public key. 
• Final Output: The resulting signature must be Base64 encoded before being placed in the 
signature header

9.6.  Get Payment Status API  
The Merchant system should verify the QR status upon receiving the WebSocket notification. If no 
response is received via WebSocket, Merchant can alternatively check the QR request status.

Example: https://{Base_URL_API}/api/merchant/third-party/v2/thirdPartyDynamicQrGetStatus 
This API is used to check the payment status of a Dynamic QR transaction using the merchant’s 
terminal ID and reference label.
